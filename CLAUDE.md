# Claude作業ルール

## 基本方針
- 作業前に必ずこのファイルを参照する
- 日本語で回答する
- 変更前に必ず現状のファイルを確認・把握してから作業を開始する
- 実行するコマンドは事前に内容を報告し、承認を得てから実行する

## ワークフロー
作業は以下の順序で行う：

1. ローカル（デスクトップ）で作業・保存　← メインの作業場所
2. 作業内容をGitにコミット（変更履歴を記録）
3. GitHubにpush（クラウドバックアップ）

```
デスクトップで作業 → コミット → GitHubへpush
```

- pushの前に必ず変更内容を報告し、承認を得てから実行する
- コミットメッセージは日本語で簡潔に書く

## ファイル・コードの変更原則

### ファイル削除の禁止
- いかなる場合もファイルの削除は行わない
- 不要になったファイルであっても削除せず、ユーザーに判断を委ねる

### バージョン管理
- コードの削除・編集を行う場合は、Gitのコミット履歴で過去バージョンを遡れるようにする
- 手動でのバージョン管理（フォルダ名やファイル名にv1.0などをつける方法）は行わない
- GitHubへのpush前に変更内容を必ず報告し、承認を得てから実行する

### 機密情報・個人情報の保護
- .envや機密情報を含むファイルは絶対にGitHubにpushしない
- APIキー・パスワード・認証情報をコードに直接書かない
- 個人情報や業務データが含まれている可能性があるファイルは、push前に必ずユーザーに確認する
  - 例：氏名・住所・連絡先・金額などが含まれるExcelやCSVファイル

## 新しいプロジェクト開始時のルール
- 新しいプロジェクトフォルダを作成したら、必ずそのフォルダ内にCLAUDE.mdを作成するようユーザーに進言する
- CLAUDE.mdに記載する内容：
  - プロジェクト概要
  - GitHubリポジトリのURL（SSH）
  - 現在の状態・経緯
  - 今後の方針
  - ファイル構成
  - 作業時の注意点
- プロジェクト内に `.claude/skills/log/SKILL.md` を作成する（Git経由で新PCにも共有される）

## 新PCセットアップ手順
Claude Codeを新しいPCで使い始める際は以下を実施する：

1. 個人用Skillを作成する
```
mkdir -p ~/.claude/skills/log
```
2. 既存プロジェクトの `.claude/skills/log/SKILL.md` の内容を `~/.claude/skills/log/SKILL.md` にコピーする
3. 各プロジェクトを `git clone` または `git pull` で取得する

### Skillの管理方針
- `~/.claude/skills/log/SKILL.md`：個人用（このPC上の全プロジェクトで使える）
- `.claude/skills/log/SKILL.md`：プロジェクト用（GitHubで共有・新PCに自動配布される）
- 両方を維持することで、どの環境でも `/log` が使えるようにする

## 作業前のリスク評価

作業を開始する前に、以下の観点でリスク評価を行いユーザーに提示する：

- **担当の判断**：各作業をClaudeが実行すべきか、ユーザーが行うべきか明示する
  - ローカル（PC上）での作業 → Claudeが実行可
  - GitHubなど外部サービスのWebサイト操作・ログイン・設定変更 → ユーザーが行う
- **リスクの明示**：取り消せない操作・機密情報に関わる操作は特に明示する
- **承認の取得**：評価結果をユーザーに提示し、進め方の合意を得てから作業を開始する

## 作業報告のルール
- 作業開始前：何をするか報告する
- 作業中：問題・疑問が生じたら即座に報告し作業を止める
- 作業完了後：何を変更したか報告する

## 核心原則
1. 最小性：必要最小限を超えない
2. 確認優先：迷ったら作業を止めて確認する
3. 透明性：失敗や問題は即座に報告する
4. 不可逆操作の禁止：削除など取り消せない操作は行わない

#### ⑦ 日直の話ミニアプリの実装

3フェーズ構成（ready → spinning → result）で実装した。
ルーレットで「今日の話題」をランダムに決めるミニアプリ。

**フェーズ①（ready）：スタート画面**
- 現在登録中のトピック一覧をチップ形式でプレビュー表示
- 「ルーレットをまわす！」ボタンでフェーズ②へ

**フェーズ②（spinning）：ルーレット演出**
- トピックを縦に並べたドラム式リスト（`.roulette-drum`）をJavaScriptで上方向にスクロールさせる疑似ルーレット
- 5周分のアイテムを生成してスクロール感を演出（`REPEAT = 5`）
- 当選アイテムを事前にランダム決定（`winIndex`）し、最終周のその位置で止まるよう移動距離を計算
- `requestAnimationFrame` ベースの手動アニメーション（加速→減速フェーズ）
  - 加速フェーズ：30フレームかけて初速8から最高速40へ線形加速
  - 減速フェーズ：残り距離が `ITEM_HEIGHT × topics.length` 以下になったら徐々に減速
- 上下に半透明のグラデーションフェード（`::before / ::after` 疑似要素）で「窓から見えている」感を演出
- 中央のハイライトライン（`.roulette-pointer`）で選択位置を強調

**フェーズ③（result）：結果表示**
- 当選トピックを大きく表示（36px太字、ピンク背景ボックス）
- `scorePopIn` アニメーションでポップイン演出
- 「もう一度回す」でフェーズ①に戻る

**管理画面（日直トピック管理）**
- トピックの追加・削除・デフォルトに戻す機能
- デフォルト5題：「今日楽しみなこと」「最近はまっていること」など
- localStorage `nichokuTopics` に保存、`nichoku-updated` カスタムイベントで本体へ通知
- Enterキーでもトピック追加が可能

**設計の判断**：
- ルーレット演出は `requestAnimationFrame` で自前実装（CSSアニメーションだと終了位置の制御が難しいため）
- トピックはデフォルト5件を用意することで、初期設定なしでも即使用可能にした

### 2026-05-12（第4セッション：歌ミニアプリの月別管理・音声ファイル対応）

#### 背景と出発点
前セッションまでで全8ミニアプリの実装が完了し、実運用フェーズに入っていた。
ユーザーから「歌を月ごとに登録・管理したい」「MP3などを直接アップロードして再生できるようにしたい」という改修要望があり、歌ミニアプリ（initSong / initAdminSong）を全面的に書き換えた。

#### データ設計の変更
- 変更前：`songMonthly = { title, url, lyrics }`（1曲のみ）
- 変更後：`songMonthlyAll = { "1": {...}, ..., "12": {...} }`（月番号をキーとするオブジェクト）
- 音声バイナリはlocalStorageの5MB制限を回避するため、IndexedDB（`MorningAssemblyDB` / ストア名 `songAudio`）に分離保存
- 旧データ（`songMonthly`）は削除せず、初回起動時に今月のエントリとして `songMonthlyAll` へ自動マイグレーション

#### IndexedDB ラッパー関数の追加（script.js）
`extractYouTubeId` より前に4関数を追加：
- `openSongAudioDB()`：DB接続・バージョン管理
- `saveSongAudio(monthKey, base64)`：音声データ保存
- `loadSongAudio(monthKey)`：音声データ読み込み
- `deleteSongAudio(monthKey)`：音声データ削除

#### initSong() の書き換え（script.js）
- ストレージキーを `songMonthly` → `songMonthlyAll` に変更
- `loadMonthly()` を「現在月のエントリを自動取得」する実装に変更
- `playSong()` を async 化：YouTube優先 → 音声ファイルフォールバックの条件分岐
- 「もう一度」ボタンに `<audio>` 停止処理を追加
- `initSong()` 先頭に旧データマイグレーション処理を追加

#### initAdminSong() の書き換え（script.js）
- 月タブ（1〜12月）をJSで動的生成し、タブ切り替えでフォームデータを更新
- 保存時は選択中の月キーで `songMonthlyAll` を更新
- MP3アップロード：FileReader で base64 変換 → IndexedDB に保存
- 音声ファイル登録済みの場合「✅ 登録済み」、未登録の場合「未登録」と表示
- 削除ボタンで IndexedDB から音声データを削除
- ファイルサイズ30MB超の場合アラートで警告
